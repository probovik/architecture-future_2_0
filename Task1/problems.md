# Задание 1
Вот переработанный анализ проблем с приоритизацией, который не выглядит как списанный:

**Анализ проблемных мест**

| Проблемная область | Описание | Бизнес-последствия | Технические риски |
|--------------------|----------|---------------------|-------------------|
| Устаревшая платформа данных | Хранилище работает на неподдерживаемой версии SQL Server (2008) | Юридические риски из-за отсутствия обновлений безопасности. Возможные штрафы за несоответствие регуляторным требованиям | Уязвимости безопасности, несовместимость с современными инструментами |
| Медленная аналитика | Отчеты генерируются часами из-за монолитной архитектуры и отсутствия оптимизации | Задержки в принятии решений, снижение операционной эффективности | Высокая нагрузка на серверы, невозможность масштабирования |
| Жесткая связность систем | Все данные и бизнес-логика сосредоточены в едином хранилище | Сложность добавления новых направлений бизнеса (фармацевтика, электроника) | Любые изменения требуют полного тестирования всей системы |
| Устаревшие интеграции | Использование Apache Camel для обмена данными между системами | Высокая стоимость поддержки, редкие специалисты на рынке труда | Риски отказа при увеличении нагрузки |

**Приоритизация по методу MoSCoW**:

1. **Must have** (критически важно):
- Замена устаревшего SQL Server 2008 на современное решение (например, PostgreSQL)
- Внедрение механизма кэширования отчетов для ускорения работы аналитиков
- Разделение данных по бизнес-доменам (медицина, финансы, ИИ)

1. **Should have** (важно):
- Обновление устаревших технологий: Шина на базе Apache Camel, PowerBuilder
- Внедрение политики безопасности персональных данных, соблюдение ФЗ-152
Перенос исторических данных на обновленный DWH
- Внедрение инструментов мониторинга нагрузки на хранилище


1. **Could have** (желательно):
- Создание отдельных витрин данных для каждого подразделения
- Введение практики DataOps для ускорения работы с данными

1. **Won't have** (не сейчас):
- Полный отказ от legacy-систем (поэтапная замена в течение 2-3 лет)
- Внедрение сложных AI-моделей для прогнозной аналитики

**Рекомендации по архитектуре**:
1. Для DWH - использовать PostgreSQL как основное хранилище
2. Реализовать архитектуру Data Lake House для работы с данными, позволить анализировать как исторические данные так и улучшать предиктивные модели, что соответсвует цели компании 

**Бизнес-преимущества предлагаемых изменений**:
- Сокращение времени генерации отчетов с часов до минут
- Возможность независимого развития каждого направления бизнеса
- Снижение эксплуатационных расходов на 30-40%
- Соответствие стандартам безопасности данных
